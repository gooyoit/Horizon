## [PostgreSQL 18 Enables Production Query Plans Without Production Data](https://simonwillison.net/2026/Mar/9/production-query-plans-without-production-data/#atom-everything) ⭐️ 9.0/10

PostgreSQL 18, released in September 2025, introduces two new functions—pg_restore_relation_stats() and pg_restore_attribute_stats()—that allow developers to export planner statistics from production and import them into development environments. This enables accurate replication of production query execution plans without copying actual production data. This feature solves a long-standing challenge where query performance issues in production were hard to reproduce locally due to differing table statistics. It significantly improves developer productivity and debugging accuracy while reducing data privacy and storage concerns associated with copying large production datasets. The statistics are compact—often under 1MB even for databases with hundreds of tables—and can be injected using function calls that specify values like null_frac, n_distinct, and most_common_vals. SQLite already offers similar functionality via writable sqlite_stat1 and sqlite_stat4 tables.

rss · Simon Willison · Mar 9, 15:05

**Background**: The PostgreSQL query planner relies on internal statistics (stored in the pg_statistic system catalog) to estimate row counts and choose optimal execution plans. These statistics are generated by ANALYZE and reflect data distribution, such as value frequencies and null ratios. Without matching statistics, a query may use an index in production but perform a full scan in development—or vice versa—leading to misleading performance testing.

## [Critical GBL Vulnerability Unlocks Snapdragon 8 Elite Gen 5 Bootloader](https://t.me/zaihuapd/40141) ⭐️ 9.0/10

Security researchers have disclosed a critical vulnerability in Qualcomm’s Snapdragon 8 Elite Gen 5 (8E5) that allows permanent bootloader unlocking by bypassing UEFI secure boot during GBL loading from the efisp partition. The exploit enables arbitrary EL1-level code execution and modification of RPMB-stored devinfo to unlock the bootloader. This vulnerability undermines the foundational security of Android devices using the latest Snapdragon flagship SoC, enabling persistent bootloader unlocking that compromises secure boot, voids warranties, and exposes devices to firmware-level malware. It also empowers rooting and modding communities but poses significant risks if exploited maliciously. The flaw exists because the Android Bootloader (ABL) fails to enforce UEFI secure boot when loading the Generic Bootloader (GBL) from the efisp partition, allowing unsigned UEFI applications to run. Researchers used this to overwrite the bootloader lock state stored in the Replay Protected Memory Block (RPMB).

telegram · zaihuapd · Mar 9, 15:20

**Background**: The Generic Bootloader (GBL) is a standardized UEFI-based bootloader component introduced by Google to streamline Android’s boot process across devices. Secure boot ensures that only cryptographically signed code runs during boot, preventing unauthorized modifications. The RPMB is a tamper-resistant storage area in eMMC/UFS used to store critical device states like bootloader lock status. Qualcomm’s ABL implementation typically enforces signature verification, but this vulnerability bypasses that protection.
